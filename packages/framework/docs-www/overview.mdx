The framework is organized around three concepts: **units** (infrastructure), **modules** (domain logic), and the **Framework** orchestrator that wires them together.

## Architecture

```text
┌──────────────────────────────────────────────────────────┐
│                    FRAMEWORK                              │
│                                                          │
│  Framework.create(config, modules)                        │
│    │                                                     │
│    ├── Validates tenancy config                          │
│    ├── Instantiates 7 units (dependency-injected)        │
│    │   db → logs → pubsub → auth → storage → kvStore → rpc │
│    │   (pubsub ↔ auth cross-wired after creation)        │
│    │                                                     │
│    └── Calls module.$initialize(units) for each module   │
│        └── Returns proxy-wrapped FrameworkInstance       │
│                                                          │
│  Proxy access:                                           │
│    framework.db          → DatabaseUnit                  │
│    framework.auth         → AuthUnit                      │
│    framework.organization → OrganizationModule           │
│    framework.getUnit("db") → DatabaseUnit                │
│    framework.getModule("organization") → OrganizationModule │
└──────────────────────────────────────────────────────────┘
```

## Entry Surfaces

The framework package (`@aspen-os/framework`) declares only `./client` and `./server` in its `exports` map. There is no root `.` entry — the bare import `@aspen-os/framework` does not resolve. Always import via subpaths: `@aspen-os/framework/server` or `@aspen-os/framework/client`.

### Server (`@aspen-os/framework/server`)

Exports:
- `Framework` class (with static `create()` factory)
- `getContext` function (access AsyncLocalStorage context inside `run()`)
- `FrameworkConfig`, `FrameworkInstance`, `FrameworkUnits` types
- `Unit`, `Module` interfaces
- `TenancyConfig`, `TenancyMode`, `TenantResolver` types
- All 7 unit classes and their config types (as types):
  - `DatabaseUnit`, `DatabaseConfig`
  - `AuthUnit`, `AuthConfig`
  - `LogUnit`, `LogConfig`
  - `PubSubUnit`, `PubSubConfig`
  - `StorageUnit`, `StorageConfig`
  - `RpcUnit`, `RpcConfig`
  - `KvStoreUnit`, `KvStoreConfig`

### Client (`@aspen-os/framework/client`)

Exports:
- Client `Framework` class (browser-side, 3 units only)
- `createAccessControl` (re-exported from `better-auth/plugins/access`)
- `FrameworkConfig`, `FrameworkInstance`, `FrameworkUnits` types
- `Unit`, `Module` interfaces
- `AuthUnit`, `AuthClient`, `LogConfig`, `RpcConfig` types

The client framework has 3 units:
- `AuthUnit` — wraps better-auth React client
- `LogUnit` — stub (throws on `$prepare()`/`$cleanup()`)
- `RpcUnit` — stub (no-op)

## FrameworkConfig

All 7 units plus a `tenancy` config are **required** in the config:

```ts title="src/server/index.ts"
type FrameworkConfig = {
  auth: AuthConfig;
  db: DatabaseConfig;
  kvStore: KvStoreConfig;
  logs: LogConfig;
  pubsub: PubSubConfig;
  rpc: RpcConfig;
  storage: StorageConfig;
  tenancy: TenancyConfig;
};
```

## Lifecycle

```text
Framework.create(config, modules)   →  creates units + initializes modules
    ↓
framework.prepare()                 →  unit.$prepare() then module.$prepare()
                                       (+ RLS policies in shared mode)
                                       (+ $prepareTenant per tenant in isolated mode)
    ↓
framework.run(fn)                   →  AsyncLocalStorage { auth, db, pubsub, tenantId? }
framework.run(tenantId, fn)         →  per-request db + tenantId in context
    ↓
framework.destroy()                 →  module.$cleanup() then unit.$cleanup()
```

### `create(config, modules)`

Static factory — the **only** way to construct a Framework. It:
1. Validates the `tenancy` config
2. Instantiates all 7 units in dependency order
3. Cross-wires `pubsub` ↔ `auth` (bidirectional dependency)
4. Calls `module.$initialize(units)` on each module
5. Returns a proxy-wrapped `FrameworkInstance`

<Callout type="warn">
  There is no `new Framework(config)` or `registerModule()`. All modules must be passed to `create()` at once.
</Callout>

### `prepare()`

Runs each unit's `$prepare()` (e.g. `DatabaseUnit.$prepare()` calls `pushSchema()`), then each module's `$prepare()`. In `shared` mode it additionally applies RLS policies; in `isolated` mode it lists tenants via the resolver and calls `$prepareTenant()` per tenant. Errors are caught and logged per-unit/module/tenant.

### `run(fn)` / `run(tenantId, fn)`

Executes `fn` inside `AsyncLocalStorage` providing `{ auth, db, pubsub, tenantId? }`:
- `db` — the drizzle `NodePgDatabase` (per-request in multi-tenant modes)
- `pubsub` — the `PubSubUnit`
- `auth` — the better-auth `Auth` instance
- `tenantId` — the tenant ID (only in multi-tenant modes)

In `single` mode, `run(fn)` is used. In `shared`/`isolated` modes, `run(tenantId, fn)` is required.

### `destroy()`

Runs `module.$cleanup()` then `unit.$cleanup()`. Errors are caught and logged.

## Dependency Graph

Units are constructor-injected:

```text
DatabaseUnit ← LogUnit
DatabaseUnit ← PubSubUnit ←→ AuthUnit  (cross-wired: pubsub.setAuth, auth.setPubSub)
DatabaseUnit ← StorageUnit
DatabaseUnit ← AuthUnit
DatabaseUnit ← KvStoreUnit
DatabaseUnit ← RpcUnit (also ← LogUnit, PubSubUnit, AuthUnit)
```

| Unit | Injected deps |
|---|---|
| `db` | — (owns `pg.Pool` + drizzle `db`) |
| `logs` | `{ db }` |
| `pubsub` | `{ db }` |
| `storage` | `{ db }` |
| `auth` | `{ db }` |
| `rpc` | `{ auth, db, logs, pubsub }` |
| `kvStore` | `{ db }` |

## Accessing Units and Modules

The `FrameworkInstance` is a Proxy that intercepts property access:

```ts
// Proxy access (preferred for readability)
framework.db              // → DatabaseUnit
framework.auth             // → AuthUnit
framework.organization     // → OrganizationModule

// Typed accessor methods
framework.getUnit("db")        // → DatabaseUnit
framework.getModule("organization") // → OrganizationModule (throws if not found)
```
