A **Module** is a package of domain business logic. Modules are optional, pluggable, and created by the application — not the framework. You choose which modules to include, create them via their static `create()` factory, and pass them to `Framework.create()`.

## The Module Interface

```ts title="src/server/index.ts"
interface Module<N extends string = string> {
  readonly name: N;
  initialize?(units: Record<string, Unit>): void;
  prepare?(): Promise<void>;
  destroy(): Promise<void>;
}
```

The interface is the same on both server and client — no `$` prefix, unlike units.

### Lifecycle methods

| Method | When it's called | What it does |
|---|---|---|
| `initialize(units)` | Inside `Framework.create()`, after all units are instantiated | Receives all 7 units. Creates workflows with `db` and `pubsub` references. |
| `prepare()` | During `framework.prepare()`, after all units have prepared | Pushes domain schemas via `pushSchema()`, registers pubsub handlers/schedules. |
| `destroy()` | During `framework.destroy()`, before units are destroyed | Nulls private fields, unregisters pubsub handlers. **Required** (not optional). |

<Callout type="warn">
  `destroy()` is the only **required** method on the `Module` interface. `initialize()` and `prepare()` are optional.
</Callout>

## How Modules Are Created

Unlike units (which the framework creates via `new`), modules are created by the application via their static `create()` factory, then passed as a named object to `Framework.create()`:

```ts
import { Framework } from "@aspen-os/framework/server";
import { OrganizationModule } from "@aspen-os/organization";
import { ComplianceModule } from "@aspen-os/compliance";

// 1. Create modules via their factory
const organization = OrganizationModule.create({ country: "INDIA" });
const compliance = ComplianceModule.create({ country: "INDIA" });

// 2. Pass them to the framework
const framework = Framework.create(config, { organization, compliance });

// 3. Prepare (runs unit.$prepare() then module.prepare())
await framework.prepare();
```

Inside `create()`, the framework iterates over the modules object and calls `mod.initialize?.(units)` on each, passing the fully-constructed `FrameworkUnits`. Module names (the `name` property) become proxy keys on the framework instance.

## Accessing Modules

### Proxy access (preferred)

```ts
framework.organization            // → OrganizationModule
framework.organization.branches   // → BranchWorkflow
framework.compliance             // → ComplianceModule
framework.compliance.documents   // → DocumentWorkflow
```

### Typed accessor

```ts
framework.getModule("organization")  // → OrganizationModule (throws if not found)
```

The proxy checks units first, then modules. If a property name matches a unit key, the unit is returned; otherwise it falls through to modules.

## Module Conventions

All domain modules in the aspen-os ecosystem follow a strict set of conventions. These are derived from the `organization` and `compliance` reference implementations.

### Static `create()` factory

Never use `new` directly. Always expose a static factory:

```ts title="src/index.ts"
export class OrganizationModule {
  static create(config: OrganizationConfig): OrganizationModule {
    return new OrganizationModule(config);
  }

  private constructor(private config: OrganizationConfig) {}
}
```

### `#` private fields with lazy getters

Workflow fields are private (`#` prefix) and initialized lazily in `initialize()`. Getters throw if accessed before initialization:

```ts
#branches: BranchWorkflow | null = null;

get branches(): BranchWorkflow {
  if (!this.#branches) throw notInitialized();
  return this.#branches;
}

initialize(units: { db: DatabaseUnit; pubsub: PubSubUnit }): void {
  this.#branches = new BranchWorkflow(units.db.db, units.pubsub);
}
```

### `initialize(units)` creates workflows

This is where domain workflows are constructed with `db` and `pubsub` references. The `units` parameter is typed as `Record<string, Unit>`, but modules typically destructure the specific units they need:

```ts
initialize(units: Record<string, Unit>): void {
  const { db, pubsub } = units;
  this.#branches = new BranchWorkflow(db.db, pubsub);
  this.#addresses = new AddressWorkflow(db.db);
}
```

### `prepare()` pushes schema

Uses `pushSchema()` from `drizzle-kit/api` to apply domain table changes:

```ts
async prepare(): Promise<void> {
  const { pushSchema } = await import("drizzle-kit/api");
  const result = await pushSchema(dbSchema, this.units.db.db);
  if (result.statementsToExecute.length > 0) {
    await result.apply();
  }
}
```

### `destroy()` nulls fields

Required. Unregisters pubsub handlers and nulls out private workflow fields to prevent use-after-destroy:

```ts
async destroy(): Promise<void> {
  this.pubsub.unsubscribe(/* ... */);
  this.#branches = null;
  this.#addresses = null;
}
```

### `db_schema` export

The module's drizzle schema namespace is exported as `db_schema`:

```ts
import * as dbSchema from "./db-schema";
export { dbSchema };
```

### `name` as kebab-case

The `name` property is a readonly string in kebab-case — this becomes the proxy key:

```ts
readonly name = "organization" as const;
```

## File Structure

```text
packages/my-module/
  src/
    index.ts              # Module class, type re-exports, dbSchema export
    db-schema.ts          # Drizzle pgTable/pgEnum definitions
    types.ts              # Type re-exports from schemas + domain interfaces
    event-map.ts          # Event constants + typed interfaces + EventMap type
    constants.ts          # Module-specific enums (as const objects) [optional]
    schemas/
      index.ts            # Re-exports all valibot schemas + types
      enums.ts            # Valibot enum schemas mirroring constants
      utils.ts            # Shared valibot schema utilities
      entity.ts           # Per-entity valibot schemas (create/update/filter)
    workflows/
      entity.ts           # Business logic classes
  package.json
  tsconfig.json
```

## Existing Modules

### Organization (`@aspen-os/organization`)

Manages organizational structure — branches, addresses, and related entities.

| Property | Workflow | Purpose |
|---|---|---|
| `branches` | `BranchWorkflow` | Create, update, list, and manage organizational branches |
| `addresses` | `AddressWorkflow` | Manage addresses associated with branches and organizations |

- **Tables**: 7 (branches, addresses, and supporting entities)
- **Config**: `{ country: CountryCode }` — determines address validation rules
- **Name**: `"organization"`

### Compliance (`@aspen-os/compliance`)

Manages compliance documents, attestations, and regulatory workflows.

| Property | Service/Workflow | Purpose |
|---|---|---|
| `documents` | `DocumentWorkflow` | Upload, review, and track compliance documents |
| `attestations` | `AttestationWorkflow` | Manage compliance attestations |

- **Services**: 3 (document, attestation, and a scheduling service)
- **Config**: `{ country: CountryCode }` — determines regulatory requirements
- **Name**: `"compliance"`

## Validation

Domain modules use **Valibot** for input validation (create/update/filter schemas). This is distinct from the RPC layer, which uses **Zod** for procedure validation.

```ts
import { type InferOutput, object, pipe, string, minLength } from "valibot";

export const CreateBranchSchema = object({
  name: pipe(string(), minLength(1, "Name is required")),
  code: pipe(string(), minLength(1, "Code is required")),
});

export type CreateBranchInput = InferOutput<typeof CreateBranchSchema>;
```

## Events

Modules define events in the `"domain:event_name"` format. Events are typed via an `EventMap` type — a compile-time contract, not a runtime bus. Events are published as plain string topics via PubSub:

```ts title="src/event-map.ts"
export const ORG_EVENTS = {
  BRANCH_CREATED: "organization:branch_created",
  BRANCH_UPDATED: "organization:branch_updated",
} as const;

export type OrganizationEventMap = {
  [ORG_EVENTS.BRANCH_CREATED]: { branch: { id: string; name: string } };
  [ORG_EVENTS.BRANCH_UPDATED]: { changes: Record<string, unknown>; branch: { id: string } };
};
```

```ts
// Publishing (inside a workflow)
await this.pubsub.publish("organization:branch_created", { branch });
```

<Callout type="info">
  The event map is a type-level contract only. PubSub accepts plain strings — the `EventMap` type ensures type safety at the call site but does not constrain what can be published at runtime.
</Callout>

## Modules vs Units

If you're unsure whether to create a unit or a module, the answer is always a **module**:

- **Units** are fixed infrastructure, created by the framework, and you cannot add new ones.
- **Modules** are pluggable domain logic, created by you, and you can add as many as you need.

See [Unit vs Module](/docs/framework/unit-vs-module) for a detailed comparison.
