The framework supports three tenancy architectures as a **config-time choice**. The developer picks one mode in `FrameworkConfig.tenancy` and commits to it for the application's lifetime. The same module code — workflows, services, schemas — works transparently across all three modes.

## Modes

| Mode | Databases | Isolation | `tenant_id` columns | Connection routing |
|---|---|---|---|---|
| **Single Tenant** | 1 | None needed | Present (always `"default"`) | Static pool |
| **Shared DB + RLS** | 1 (shared) | Postgres RLS policies | Present (varies per row) | Per-request client + `SET LOCAL` |
| **Isolated DB** | N+1 (control-plane + per-tenant) | Physical (separate DBs) | Present (redundant per DB) | Per-tenant pool resolution |

## Configuration

The `tenancy` field is **required** on `FrameworkConfig`:

```ts
import type { TenancyConfig } from "@aspen-os/framework/server";

// Single tenant
const tenancy: TenancyConfig = { mode: "single" };

// Shared DB with Row-Level Security
const tenancy: TenancyConfig = { mode: "shared" };

// Isolated database per tenant
const tenancy: TenancyConfig = {
  mode: "isolated",
  resolver: {
    resolve: async (tenantId) => ({ /* DatabaseConfig */ }),
    list: async () => ["tenant_1", "tenant_2"],
  },
};
```

### Full example

```ts
const framework = Framework.create(
  {
    auth: authConfig,
    db: dbConfig,          // control-plane DB (the app DB in single/RLS mode)
    kvStore: kvConfig,
    logs: logConfig,
    pubsub: pubsubConfig,
    rpc: rpcConfig,
    storage: storageConfig,
    tenancy: { mode: "single" },
  },
  { organization },
);
```

The `db` config is always the **control-plane** database. In single/RLS mode, this IS the app database. In isolated mode, this is the control-plane database (auth + platform tables); per-tenant DB configs are resolved by the `TenantResolver`.

## `run()` Signatures

```ts
// Single-tenant mode
await framework.run(async () => {
  const { db } = getContext();
  // db resolves to the control-plane (app) database
});

// Multi-tenant modes (shared or isolated)
await framework.run(tenantId, async () => {
  const { db, tenantId } = getContext();
  // db resolves to the per-request/per-tenant drizzle instance
});
```

The caller resolves the tenant ID (e.g., from the auth session's `activeOrganizationId`) and passes it to `run()`. The framework doesn't know about auth — it just takes a tenant ID.

## How It Works

### Stable DB Wrapper

`DatabaseUnit.db` is a getter that returns a stable wrapper (a JavaScript `Proxy`). The wrapper resolves the correct per-request drizzle instance via `AsyncLocalStorage` at query time. Workflows keep `this.db = units.db.db` — no workflow code changes.

- **Single-tenant**: context db is the control-plane db. Falls back to control-plane db outside `run()`.
- **shared**: `run(tenantId, fn)` acquires a dedicated client, starts a transaction, sets `SET LOCAL app.tenant_id`, creates a drizzle instance wrapping that client. The wrapper delegates to it.
- **isolated**: `run(tenantId, fn)` resolves the per-tenant drizzle instance. The wrapper delegates to it.

### Control-Plane Connection

`DatabaseUnit` always holds a control-plane connection. `AuthUnit` always uses `controlPlaneDb` — auth tables are only on the control-plane DB and are exempt from RLS.

```ts
framework.db.controlPlaneDb   // drizzle instance for control-plane queries
framework.db.pool              // pg.Pool for the control-plane connection
```

### Per-Tenant Pool Management (isolated)

```ts
framework.db.getTenantDb(tenantId)       // lazily creates + caches per-tenant pool
framework.db.pushSchemasToTenant(tenantId, moduleSchemas)  // provisioning
```

### RLS Policies (shared Mode)

After all schemas are pushed during `prepare()`, the framework discovers all tables with a `tenant_id` column and applies standard RLS policies:

```sql
ALTER TABLE <table> ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON <table>
  FOR ALL TO tenant_role
  USING (tenant_id = current_setting('app.tenant_id', true))
  WITH CHECK (tenant_id = current_setting('app.tenant_id', true));
```

The framework creates a `tenant_role` Postgres role and grants it to the connection user. `run(tenantId, fn)` sets `SET LOCAL ROLE tenant_role` so RLS applies to per-request queries, while control-plane queries (as the table owner) bypass RLS.

### `tenant_id` Column

Every table (except auth tables) gains a `tenant_id` column:

```ts
tenantId: text("tenant_id")
  .notNull()
  .default(sql`COALESCE(current_setting('app.tenant_id', true), 'default')`)
```

- **Single-tenant**: always `"default"` (the GUC is never set).
- **shared**: set per-request via `SET LOCAL app.tenant_id`.
- **isolated**: `"default"` (redundant but harmless — isolation is physical).

### Per-Tenant PubSub (isolated)

Each tenant DB has its own pg-boss instance. `PubSubUnit` routes `publish`/`subscribe`/`schedule` to the correct per-tenant pg-boss based on the context's `tenantId`. A control-plane pg-boss handles platform-level events.

Use `publishControlPlane()` for control-plane events (e.g., auth events):

```ts
framework.pubsub.publishControlPlane("user:created", { user });
```

### `$prepareTenant()` Lifecycle

The `Module` interface gains an optional `$prepareTenant(tenantId)` method. In isolated mode, the framework calls it for each tenant at startup and during provisioning. Modules register per-tenant cron schedules and subscriptions here.

```ts
interface Module<N extends string = string> {
  readonly $name: N;
  $initialize?(units: Record<string, Unit>): void;
  $prepare?(): Promise<void>;
  $prepareTenant?(tenantId: string): Promise<void>;  // isolated mode
  $destroy(): Promise<void>;
}
```

### Tenant-Aware Storage & KV

- **Storage**: S3 keys are prefixed with `tenantId`. `file_metadata` has a `tenant_id` column.
- **KV Store**: Keys are prefixed with `tenantId`. `kv_store` has a `tenant_id` column.
- **Logs**: Log entries include `tenant_id`, scoped from context.

## CLI

### `db-studio` with `--tenant`

```bash
# Control-plane database
aspen db-studio --config=src/aspen/server.ts

# Per-tenant database (isolated mode)
aspen db-studio --config=src/aspen/server.ts --tenant=tenant_123
```

### `tenants` command

```bash
aspen tenants --config=src/aspen/server.ts
```

Lists all tenant IDs via `resolver.list()` (isolated mode only).

## Migration

### Existing Apps

Add `tenancy: { mode: "single" }` to `FrameworkConfig`. The `tenant_id` column is added automatically by `pushSchema()`. No other changes needed — workflows work as before.

### New Multi-Tenant Apps

1. Choose a mode: `shared` or `isolated`.
2. Configure `tenancy` in `FrameworkConfig`.
3. In RLS mode: the framework creates the `tenant_role` and applies policies during `prepare()`.
4. In isolated mode: provide a `TenantResolver`.
5. Resolve `tenantId` from the auth session and pass to `run(tenantId, fn)`.
6. Register modules as before — they work transparently.
