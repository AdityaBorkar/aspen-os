The HR module exposes 7 workflow classes, each managing a distinct HR domain aggregate.

## EmployeeWorkflow

Access: `framework.hr.employees`

Manages employee records, employee groups, health insurance, and skill maps.

### Employee CRUD

```ts
create(input: CreateEmployeeInput)
update(id: string, patch: UpdateEmployeeInput)
getById(id: string)
getByEmployeeId(employeeId: string)
list(filters?: EmployeeFilters)
deactivate(id: string)
activate(id: string)
markAsLeft(id: string, dateOfLeaving: string)
getOrganizationalChart(company?: string): Promise<EmployeeTreeNode[]>
```

### Employee Group

```ts
createGroup(input: CreateEmployeeGroupInput)
updateGroup(id: string, patch: UpdateEmployeeGroupInput)
getGroupById(id: string)
listGroups()
deleteGroup(id: string)
addGroupMember(input: AddGroupMemberInput)
removeGroupMember(groupId: string, employeeId: string)
listGroupMembers(groupId: string)
```

### Health Insurance

```ts
createHealthInsurance(input: CreateHealthInsuranceInput)
updateHealthInsurance(id: string, patch: UpdateHealthInsuranceInput)
getHealthInsuranceById(id: string)
listHealthInsuranceByEmployee(employeeId: string)
deleteHealthInsurance(id: string)
```

### Skill Map

```ts
createSkillMap(input: CreateSkillMapInput)
updateSkillMap(id: string, patch: UpdateSkillMapInput)
getSkillMapById(id: string)
listSkillMapByEmployee(employeeId: string)
deleteSkillMap(id: string)
```

### Business Rules

- Employee ID uniqueness enforced on `create` and `update`
- `deleteGroup` removes all members before deleting the group
- `addGroupMember` verifies employee and group exist, rejects duplicate membership
- `createHealthInsurance` and `createSkillMap` verify the employee exists before creating

### Checkin


### Attendance Request

```ts
createAttendanceRequest(input: CreateAttendanceRequestInput)
updateAttendanceRequest(id: string, patch: UpdateAttendanceRequestInput)
```

### Business Rules

- `createLeaveApplication` checks the block list and leave balance (skipped when `isLeaveWithoutPay` is `true`)
- `approveLeaveApplication` increments allocation `usedDays` and creates a ledger entry with `transactionType: "application"`
- `cancelLeaveApplication` decrements allocation `usedDays` (floored at 0) and creates a reversal ledger entry with `transactionType: "cancellation"`
- `approveCompensatoryLeave` creates a leave allocation and a `transactionType: "compensatory"` ledger entry
- `deleteLeaveType` and `deleteLeavePeriod` are soft deletes (`isActive: false`)
- `deleteLeaveAllocation` sets `status: "cancelled"`
- `deleteLeavePolicy` hard deletes, cascading to policy details first
- `createLeavePolicyDetail` verifies both the leave policy and leave type exist
- `createLeavePolicyAssignment` verifies both the leave policy and leave period exist
- `createLeaveAllocation` verifies the leave type and leave period exist

## OvertimeWorkflow

Access: `framework.hr.overtime`

Manages overtime types and overtime slips with approval workflow and summary reporting.

### Overtime Type

```ts
createOvertimeType(input: CreateOvertimeTypeInput)
updateOvertimeType(id: string, patch: UpdateOvertimeTypeInput)
getOvertimeTypeById(id: string)
listOvertimeTypes()
deleteOvertimeType(id: string)
```

### Overtime Slip

```ts
createOvertimeSlip(input: CreateOvertimeSlipInput)
updateOvertimeSlip(id: string, patch: UpdateOvertimeSlipInput)
getOvertimeSlipById(id: string)
listOvertimeSlips(filters?: OvertimeSlipFilters)
approveOvertimeSlip(id: string, approvedBy: string)
rejectOvertimeSlip(id: string, rejectedBy: string, rejectionReason: string)
getOvertimeSummary(employeeId: string, fromDate: string, toDate: string): Promise<OvertimeSummary>
deleteOvertimeSlip(id: string)
```

### Business Rules

- `approveOvertimeSlip` calculates `amount` using the overtime type's multipliers: `standardHours × rate × standardMultiplier + holidayHours × rate × holidayMultiplier + weekendHours × rate × weekendMultiplier` (only when `amountCalculation === "fixed"` and `fixedHourlyRate` is set)
- `getOvertimeSummary` aggregates only approved slips within the date range
- `deleteOvertimeType` is a soft delete (`isActive: false`)
- `createOvertimeSlip` verifies the overtime type exists

### Business Rules

- `updateHrSettings` and `updatePayrollSettings` are upserts — they create the singleton settings row if it doesn't exist
- Department codes are uppercased on `create` and `update`
- `createDepartment` validates the parent department exists when `parentDepartment` is provided
- `updateDepartment` prevents circular references by walking the parent chain (max depth 10)
- A department cannot be set as its own parent
- `updateDepartment` enforces department code uniqueness on update
- `deleteEmploymentType`, `deleteDepartment`, `deleteDesignation`, `deleteEmployeeGrade`, `deleteHolidayList` are soft deletes (`isActive: false`)
- `deleteHoliday` is a hard delete
- `createHoliday` verifies the holiday list exists
