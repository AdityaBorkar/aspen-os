A **Unit** is an infrastructure service managed by the platform. There are exactly seven on the server and three on the client. You never instantiate units yourself — you provide configuration and the platform wires them together with dependency injection.

## The Unit Interface

### Server

```ts title="src/server/index.ts"
interface Unit {
  readonly $name: string;
  $cleanup(): Promise<void>;
  $prepare?(): Promise<void>;
}
```

Server units use a `$` prefix on lifecycle methods (`$name`, `$prepare`, `$cleanup`). This deliberately separates framework lifecycle calls from the unit's own public API — for example, `DatabaseUnit.$cleanup()` shuts down the connection pool, while `DatabaseUnit.db` is the drizzle instance you use day-to-day.

  readonly $name: string;

## The Seven Server Units

All seven are **required** in `PlatformConfig`. They are instantiated in dependency order inside `Platform.create()`:

| Key | Unit class | `$name` | Depends on | Purpose |
|---|---|---|---|---|
| `db` | `DatabaseUnit` | `"db"` | — | PostgreSQL connection pool + Drizzle ORM |
| `logs` | `LogUnit` | `"logs"` | `{ db }` | Structured logging to the database |
| `pubsub` | `PubSubUnit` | `"pubsub"` | `{ db }` | Pub/sub messaging, subscriptions, scheduled jobs |
| `storage` | `StorageUnit` | `"storage"` | `{ db }` | S3-compatible file storage with signed URLs |
| `auth` | `AuthUnit` | `"auth"` | `{ db }` | Authentication via better-auth (sessions, users, roles, 2FA, passkeys) |
| `rpc` | `RpcUnit` | `"rpc"` | `{ auth, db, logs, pubsub }` | oRPC procedure registry and HTTP handler |
| `kvStore` | `KvStoreUnit` | `"kvStore"` | `{ db }` | Key-value cache with TTL, counters, getOrSet |

### Dependency Graph

```text
db ← logs
db ← pubsub
db ← storage
db ← auth
db ← rpc (also ← logs, pubsub, auth)
db ← kvStore
```

`DatabaseUnit` is the root — every other unit depends on it, either directly or transitively. `RpcUnit` is the most connected, pulling in four other units.

## The Three Client Units

The client framework (`@aspen-os/platform/client`) instantiates only three units:

| Key | Unit class | Purpose |
|---|---|---|
| `auth` | `AuthUnit` | Wraps the better-auth React client |
| `logs` | `LogUnit` | Stub — throws on `$prepare()` / `$cleanup()` |
| `rpc` | `RpcUnit` | Stub — no-op |

Client units receive no constructor-injected dependencies. The client `AuthUnit` takes the `access_control` and `roles` config and passes them to the `adminClient()` plugin for client-side role checking.

## How Units Are Created

You never call `new DatabaseUnit(...)` yourself. You provide a `PlatformConfig` object and the platform does the rest:

```ts
const platform = Platform.create(
  {
    db: { host, port, user, password, database },
    auth: { baseURL, secret, access_control, roles, ... },
    logs: { defaultLevel: "info", serviceName: "my-app" },
    pubsub: {},
    storage: { bucket, prefix, provider: { ... } },
    rpc: { prefix: "/api/rpc" },
    kvStore: { defaultTtl: 3600, keyPrefix: "app" },
    tenancy: { mode: "single" },
  },
  { organization, compliance },
);
```

Inside `create()`, the platform instantiates each unit with `new` in dependency order, passing previously-created units as constructor arguments. The resulting `PlatformUnits` object is then passed to each module's `$initialize(units)`.

<Callout type="warn">
  The unit set is fixed. You cannot add, remove, or replace units. All seven configs are required — there are no optional units.
</Callout>

## Unit Lifecycle

Units participate in the platform's `prepare()` and `destroy()` phases:

```text
platform.prepare()
  → for each unit: unit.$prepare?.()   // catches + logs per-unit errors
  → for each module: mod.$prepare?.()
  → shared mode: applyRlsPolicies()
  → isolated mode: $prepareTenant(tenantId) per tenant

platform.destroy()
  → for each module: mod.$cleanup()     // catches + logs per-module errors
  → for each unit: unit.$cleanup()      // catches + logs per-unit errors
```

### `$prepare()`

Called during `platform.prepare()`. This is where a unit sets up its runtime state:

- `DatabaseUnit.$prepare()` pushes core database schemas (auth, logs, storage, kv-store) via `pushSchema()` from `drizzle-kit/api`.
- `PubSubUnit.$prepare()` starts the control-plane pg-boss instance.
- `AuthUnit.$prepare()` is a no-op (auth tables are pushed by `DatabaseUnit`).
- Other units may override `$prepare()` if they need async initialization.

Errors are caught and logged per-unit — a failing unit does not prevent other units or modules from preparing.

### `$cleanup()`

Called during `platform.destroy()`, after all modules are destroyed. This is where a unit releases resources:

- `DatabaseUnit.$cleanup()` ends the `pg.Pool`.
- `PubSubUnit.$cleanup()` stops the monitor loop and unregisters listeners.
- Other units close connections, clear caches, or release handles.

Errors are caught and logged — a failing destroy does not prevent other units from being destroyed.

## Schema Ownership

Each server unit that needs persistence owns its own database schema. These are pushed automatically during `DatabaseUnit.$prepare()`:

| Unit | Tables |
|---|---|
| `auth` | `user`, `session`, `account`, `verification` |
| `logs` | `logs` |
| `storage` | `file_metadata` |
| `kvStore` | `kv_store` |

Domain modules push their own schemas in their `prepare()` method — the platform does not manage module schemas.

## Accessing Units

The `PlatformInstance` is a `Proxy` that intercepts property access. Units are available via two patterns:

### Proxy access (preferred)

```ts
platform.db       // → DatabaseUnit
platform.auth     // → AuthUnit
platform.pubsub   // → PubSubUnit
platform.storage  // → StorageUnit
platform.kvStore  // → KvStoreUnit
platform.logs     // → LogUnit
platform.rpc      // → RpcUnit
```

### Typed accessor

```ts
platform.getUnit("db")     // → DatabaseUnit
platform.getUnit("auth")   // → AuthUnit
```

The proxy checks units first, then modules. If a property name matches a unit key, the unit is returned; otherwise it checks modules; otherwise it falls through to the `Platform` class's own methods.

### Inside request context

Within `platform.run(fn)`, the drizzle `db` instance, `pubsub`, `auth`, and `tenantId` are available via `AsyncLocalStorage`:

```ts
import { getContext } from "@aspen-os/platform/server";

await platform.run(async () => {
  const { auth, db, pubsub } = getContext();
  // db: NodePgDatabase
  // pubsub: PubSubUnit
  // auth: better-auth Auth instance
});

// Multi-tenant
await platform.run(tenantId, async () => {
  const { db, tenantId } = getContext();
});
```

## Why You Can't Create Custom Units

Units represent fixed infrastructure — database, auth, storage, pubsub. These are framework-level concerns with well-defined contracts. The set is closed by design:

- **Stable contracts**: Each unit's constructor signature and injected dependencies are known at compile time. The platform's `PlatformUnits` type is a fixed map.
- **Dependency order**: Instantiation order matters (units depend on each other). The platform controls this via `Platform.create()`.
- **Lifecycle guarantees**: The platform calls `$prepare()` and `$cleanup()` in the correct order, with error isolation per-unit.

If you need to add functionality, create a **module** instead — modules are pluggable, receive all units via `initialize()`, and follow their own lifecycle. See [Modules](/docs/platform/module) and [Custom Module](/docs/platform/custom-module).
